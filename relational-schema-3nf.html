Relational database schema (third normal form)

Logical schemas derived from the DonorBridge ERD by mapping entities and relationships, then normalizing to third normal form (3NF): each table has atomic columns, no partial dependencies on composite keys, and no transitive dependence of non-key attributes on the primary key through other non-key attributes.

1. Entity–relationship → relational mapping

Each strong entity becomes a relation with the same attributes. Weak or optional subtypes are expressed with foreign keys to the supertype primary key.

2. Normalization process

2.1 Unnormalized / initial logical grouping

Conceptually group repeating or composite values out of narratives: donations contain many units, requests may reference both blood-type needs and organ needs, chats contain many messages. The ERD already separates these into entity sets instead of burying repeating groups inside PATIENT or REQUEST, which would violate 1NF.

1NF: Every relation has atomic column values (no repeating groups inside a cell). Separate relations BloodDonation, BloodUnit, ChatMessage, etc., satisfy atomicity by construction.

2.2 Functional dependencies

Hospital
hospital_id → name, location, contact — single determinant (PK).
Patient, Donor, Medical records
Natural identifiers {patient_id}, {donor_id}, {record_id} determine all non-key attributes.
Request subtree
request_id determines core request facts; subtype tables extend the same PK with mutually exclusive extension rows keyed by request type policy (enforce in application or with CHECK / assertions).
Blood unit
blood_unit_id → blood_donation_id, blood_group, volume_ml, expiry_date, unit_status. No subset of a composite PK is involved (simple PK ⇒ automatically 2NF).
Blood inventory (critical)
A column such as “available_units_summary” is not functionally determined only by (hospital_id, blood_group) in a stable way if it is refreshed from other facts; it is a derived aggregate. Storing it alongside hospital_id, blood_group creates update anomalies relative to BloodUnit.
2NF: All relations use either a single-attribute PK or, where a composite PK exists, every non-key attribute depends on the whole key. All listed relations with simple PKs are 2NF. Subtype tables BloodRequestDetails and OrganRequestDetails use full request_id as PK — no partial dependencies.
3NF: No non-key attribute may depend transitively on the primary key through another non-key attribute. Relative to the raw ERD, three concrete rules keep the logical model in 3NF:
  • Blood inventory: Do not store derived counts (for example available_units_summary) on the inventory row. Compute counts from BloodUnit in a view or in application logic. The base table stores only facts about the hospital-and-group slot (for example identifiers and last_updated).
  • Intent and SQL template: Store intent_code only on SqlTemplate. IntentDetection refers to the template with template_id; do not repeat intent_code on detection rows.
  • Query execution log: Reference the detection with intent_id only. Template text and intent_code are reached by joining IntentDetection to SqlTemplate; storing template_id again on the log would repeat a fact already determined by intent_id.

The relation list in the next section is written to satisfy these rules. Each table is in third normal form (3NF).

6. Constraints

Below are integrity and consistency rules expressed as database-style constraints where possible. Anything left to application logic should still be enforced in one place only.

6.1 Key and uniqueness

Scope Constraint
All tables Every table has exactly one declared primary key as listed in §4.
SqlTemplate UNIQUE (intent_code) — one active template catalog row per intent code when combined with business rules.
BloodInventoryLocation UNIQUE (hospital_id, blood_group) — at most one inventory row per hospital and ABO grouping.

6.2 Referential constraints

All relationships in §5 use foreign keys with semantics appropriate to optional links (nullable when the ERD allows zero or one). Use ON DELETE RESTRICT (or NO ACTION) for parent deletes unless a controlled cascade is intentional (for example cascading message deletes inside a chat session).

6.3 Request supertype and subtypes

Rule Enforcement idea
Subtype consistency If Request.request_type is BLOOD, a matching row exists in BloodRequestDetails and none in OrganRequestDetails; if ORGAN, the reverse. CHECK, trigger, or single transaction in application layer.

6.4 Match candidates (blood versus organ)

Rule Enforcement idea
Keyed resource matches match_type If match_type is a blood match: blood_unit_id IS NOT NULL and organ_offer_id IS NULL. If organ match: organ_offer_id IS NOT NULL and blood_unit_id IS NULL. Express with CHECK or assert in a stored procedure.
Score and status CHECK (compatibility_score BETWEEN 0 AND 1) if scores are normalized; otherwise use your domain range. Restrict match_status to a fixed code set via CHECK or lookup table.

6.5 Domains and non-negativity

Relation / column Constraint
Patient.age, Donor.age CHECK (age >= 0 AND age <= 150) (or link to date of birth only).
BloodDonation.quantity_donated_ml CHECK (quantity_donated_ml >= 0)
BloodUnit.volume_ml CHECK (volume_ml >= 0)
BloodRequestDetails.units_required, Request.urgency_level CHECK (units_required > 0); urgency in allowed range or enum.
IntentDetection.confidence_score CHECK (confidence_score BETWEEN 0 AND 1).
QueryExecutionLog.rows_returned CHECK (rows_returned >= 0).
Dates BloodUnit.expiry_date not before donation date when both are known; optional CHECK or trigger.

6.6 Query execution log

Optional foreign keys to Request, MatchCandidate, and BloodInventoryLocation should reflect the actual context of the executed safe query: at most one primary business object per log row, or document a JSON convention in param_json if multiple references are required.
